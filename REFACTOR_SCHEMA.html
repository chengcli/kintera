kintera × kinetics-base-titan — Refactor Schema

Drafted 2026-05-19 · Branch ions-support-plan · Best matched: 159/531 (30%), cation@L30 = 0.83× KB · Top of branch: 302c94d

One-sentence goal. Split the codebase along a physics-generic vs Titan/KB-specific seam so that (a) the chemistry/transport solver in python/atm2d/ works on any ion-bearing atmosphere without knowing about KINETICS-base file formats, and (b) python/kinetics_base_titan/ shrinks into a thin adapter that loads KB inputs and exposes Titan-specific physics knobs.

core (kintera) Titan/KB-specific currently entangled — needs splitting

1. Rationale

This session closed the biggest cation-balance gaps (cation@L30 went from 16+ OoM → 0.83× KB) by repeatedly touching code that lives on both sides of the kintera/kinetics_base_titan boundary — charge-balance Jacobian fold, atomic-budget projection, dt-cap schedule, cold-trap pin. Today most "fix-one-thing" PRs end up mutating both layers, which makes it hard to:

  • Use the kintera solver for a non-KB workflow (e.g., a new Jupiter ion run that doesn't read .pun files).
  • Test the solver in isolation (kintera unit tests today need a KB .pun file to construct a state).
  • Reason about KB-correctness vs solver-correctness (G19/G26 was a KB-semantic fix; G18b/G22 was a solver fix; they had to be teased apart by reading commits).

The remaining gaps (ambipolar diffusion, NH₃ low-altitude feedback, grain implementation bugs, HCN sawtooth) all sit cleanly on one side of the seam once the seam is explicit. Drawing the line now keeps that work from re-blurring it.

2. Three-Layer Model

LayerConcernExamplesOwner
L1. Solver core Generic atmospheric chemistry+transport math. Knows about species, concentrations, Jacobians, BCs — not about file formats or planets. Backward-Euler Newton, transport assembly, mass cap, atomic projection, charge-balance fold, dt schedule controller python/atm2d/
L2. KB adapter Parse KB (Fortran) input files into kintera state. Translate KB run-input semantics (FREEZE, SUBLIM, NCYCLE, NFIX, …) into kintera's L1 knobs. .pun / .special / .bc_save parsers, atm-file loader, KB schedule reproduction, Titan grain source classes python/kinetics_base_titan/
L3. Driver Wire L1 + L2 together for a specific experiment. Sets env-var knobs, dumps state, runs comparisons. Stability runs, NT sweeps, rate diff, trajectory compare diagnostics/

7. Backwards-Compat Strategy

The refactor touches many import paths. To avoid breaking diagnostics in-flight, propose a phased rollout:

  1. Phase 1 — Pure extractions (no behavior change)
    • Lift atomic projection, charge_balance fold, mass cap, stage schedule into their new homes and re-export from old locations.
    • Add unit tests for each extracted symbol — these have no tests today.
    • Diff kt_g29_full_dt1e7_100.npz before/after to verify bit-identical numerical behavior.
  2. Phase 2 — Rename kinetics_base_titan → kinetics_base with titan/ submodule
    • Keep kinetics_base_titan as a shim package re-exporting from kinetics_base.titan.
    • Update diagnostics/ and kintera/__init__.py re-exports.
  3. Phase 3 — Generic pin mask API
    • Replace kinetics_base_titan_boundary_pin_mask with a BoundaryPinSpec-driven equivalent.
    • Cheng cold-trap becomes a BoundaryPinSpec factory in kinetics_base/titan/cold_trap.py.
    • Diagnostics get a smaller surface area.
  4. Phase 4 — Solver decomposition
    • Split atm2d/source.py and atm2d/newton.py into the proposed sub-packages.
    • Internal-only refactor; public symbols re-exported from atm2d/__init__.py.
  5. Phase 5 — Sub-cycling / operator-split as a first-class L1 API
    • Move the driver loop from no_grain_stability.py into atm2d/newton/operator_split.py.
    • Drivers in diagnostics/ become <100-LOC config wiring.
  6. Phase 6 — New physics (ambipolar diffusion, type-4 BC, grain fix)
    • Now landing into a cleanly separated codebase; PRs touch one layer at a time.
